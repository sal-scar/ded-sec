Backups in Termux: Save Your Projects in 10 Minutes

A lightweight backup plan for your Termux work: what to back up, where to store it, and how to restore fast.

Updated: 2026-01-11 • Maintenance & reliability

Why you can trust this guide

It focuses on simple, offline-friendly methods (archives + clear folder conventions) that work on low-end phones.

1) What to back up (and what to ignore)

Back up your work, not your cache: ~/Projects, your notes, and important configs. You can reinstall packages later.

Avoid backing up gigantic folders like node_modules or Python venvs unless you truly need them.

  • ~/Projects (code + content)
  • ~/notes / README / docs
  • ~/.ssh (keys) — keep private!

2) One-command backup (tar.gz)

A tar.gz archive is portable and fast. Store it in Downloads or on external storage.

termux-setup-storage

tar -czf /storage/emulated/0/Download/termux-projects-$(date +%Y-%m-%d).tar.gz ~/Projects

3) Verify your backup (don’t guess)

Always test that the archive can be listed and that key folders exist.

tar -tzf /storage/emulated/0/Download/termux-projects-*.tar.gz | head

4) Restore after reinstall

Reinstall Termux, grant storage, then extract the archive back to ~.

termux-setup-storage
cd ~
tar -xzf /storage/emulated/0/Download/termux-projects-2026-01-11.tar.gz

5) A routine you can actually keep

Weekly: update packages. Monthly: make one archive. Before big changes: make an archive first.

  • Weekly: pkg update && pkg upgrade
  • Monthly: tar backup to Downloads
  • Before experiments: backup first

What to back up first (so you don’t regret it)

Backups fail when they are too big, too complicated, or never tested. Start with the files that would hurt to lose.

Priorities for most Termux users:

  • ~/.ssh (keys and config) — protect with permissions
  • ~/.config (app settings)
  • Your projects folder (code + notes)
  • Exported Termux packages list (so you can rebuild fast)

A simple 3-2-1 plan (without fancy tools)

3-2-1 means: 3 copies of important data, 2 different media, 1 off-site. You can do this with a USB drive + cloud + your phone.

The key is automation: schedule it, and keep it predictable.

  • Daily: small sync of projects to cloud (encrypted if possible).
  • Weekly: copy the full projects folder to USB/PC.
  • Monthly: test-restore one folder to confirm it works.

Test your restore (the step everyone skips)

A backup you can’t restore is not a backup. Do a quick restore test while everything is calm—so you’re not learning under stress after a crash.

Quick restore test idea: copy the backup to a new folder and run your project from there. If it runs, you’re safe.

  • Check file permissions (especially for .ssh).
  • Confirm key files open correctly (notes, configs).
  • Document the restore steps in a README-for-you.
